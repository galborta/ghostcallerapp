Meditation App Backend Structure

This document outlines the backend architecture for the Meditation App using Supabase. It covers database schema, authentication logic, storage structure, security policies, and implementation guidelines.

Database Schema

The database schema consists of several PostgreSQL tables with relationships that support the app's core functionality including user authentication, meditation sessions, artist profiles, and the referral system.

Core Tables

1. users Table

Extends Supabase Auth with additional user profile information:

CREATE TABLE users (

  id UUID PRIMARY KEY REFERENCES auth.users(id),

  email TEXT NOT NULL UNIQUE,

  display_name TEXT,

  avatar_url TEXT,

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  last_login TIMESTAMP WITH TIME ZONE DEFAULT NOW(),

  notification_enabled BOOLEAN DEFAULT TRUE,

  total_meditation_minutes INTEGER DEFAULT 0,

  longest_streak INTEGER DEFAULT 0,

  current_streak INTEGER DEFAULT 0,

  last_meditation_date DATE,

  referring_artist_id UUID REFERENCES artists(id),

  discount_applied BOOLEAN DEFAULT FALSE,

  discount_expiry_date DATE,

  referral_code TEXT UNIQUE,

  referral_count INTEGER DEFAULT 0,

  discount_percentage INTEGER DEFAULT 0,

  is_superfan BOOLEAN DEFAULT FALSE

);

2. artists Table

Stores information about meditation content creators:

CREATE TABLE artists (

  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

  name TEXT NOT NULL,

  bio TEXT NOT NULL,

  short_bio TEXT NOT NULL,

  image_url TEXT NOT NULL,

  featured BOOLEAN DEFAULT FALSE,

  revenue_share_percentage INTEGER DEFAULT 50,

  referral_code TEXT UNIQUE NOT NULL,

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL

);

3. meditation_tracks Table

Contains meditation audio content:

CREATE TABLE meditation_tracks (

  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

  title TEXT NOT NULL,

  artist_id UUID REFERENCES artists(id) NOT NULL,

  audio_url TEXT NOT NULL,

  audio_storage_path TEXT NOT NULL,

  duration INTEGER NOT NULL, -- in seconds

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL

);

4. meditation_sessions Table

Records user meditation activity:

CREATE TABLE meditation_sessions (

  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

  user_id UUID REFERENCES users(id) NOT NULL,

  track_id UUID REFERENCES meditation_tracks(id) NOT NULL,

  duration INTEGER NOT NULL, -- actual session duration in seconds

  is_guided BOOLEAN NOT NULL DEFAULT FALSE, -- track whether user chose guided or music-only

  completed BOOLEAN NOT NULL DEFAULT TRUE,

  completed_at TIMESTAMP WITH TIME ZONE NOT NULL,

  date DATE NOT NULL, -- extracted date for easier querying

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  -- RLS policies will ensure users can only see their own sessions

  CONSTRAINT unique_user_session UNIQUE(user_id, track_id, completed_at)

);

5. user_referrals Table

Tracks user referral relationships:

CREATE TABLE user_referrals (

  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

  user_id UUID REFERENCES users(id) NOT NULL,

  referrer_id UUID REFERENCES users(id),

  artist_id UUID REFERENCES artists(id),

  referral_date TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  discount_applied BOOLEAN DEFAULT TRUE,

  subscription_active BOOLEAN DEFAULT FALSE,

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  CONSTRAINT unique_user_referral UNIQUE(user_id, COALESCE(referrer_id, uuid_nil()), COALESCE(artist_id, uuid_nil()))

);

6. artist_earnings Table

Tracks artist revenue from referrals:

CREATE TABLE artist_earnings (

  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

  artist_id UUID REFERENCES artists(id) NOT NULL,

  year INTEGER NOT NULL,

  month INTEGER NOT NULL,

  total_referred_users INTEGER NOT NULL DEFAULT 0,

  active_subscribers INTEGER NOT NULL DEFAULT 0,

  earnings_amount DECIMAL(10,2) NOT NULL DEFAULT 0,

  is_paid BOOLEAN DEFAULT FALSE,

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,

  CONSTRAINT unique_artist_month UNIQUE(artist_id, year, month)

);

Database Indexes

Create appropriate indexes to optimize query performance:

-- Optimize lookup of user's meditation sessions

CREATE INDEX idx_meditation_sessions_user_date ON meditation_sessions(user_id, date);

 

-- Optimize artist meditation track lookup

CREATE INDEX idx_meditation_tracks_artist ON meditation_tracks(artist_id);

 

-- Optimize referral lookups

CREATE INDEX idx_user_referrals_referrer ON user_referrals(referrer_id);

CREATE INDEX idx_user_referrals_artist ON user_referrals(artist_id);

 

-- Optimize artist earnings queries

Database Triggers

Implement triggers for maintaining data integrity:

-- Update users.updated_at whenever a record is modified

CREATE TRIGGER update_users_updated_at

BEFORE UPDATE ON users

FOR EACH ROW

EXECUTE FUNCTION update_updated_at_column();

 

-- Update artists.updated_at whenever a record is modified

CREATE TRIGGER update_artists_updated_at

BEFORE UPDATE ON artists

FOR EACH ROW

EXECUTE FUNCTION update_updated_at_column();

 

-- Helper function for the above triggers

CREATE OR REPLACE FUNCTION update_updated_at_column()

RETURNS TRIGGER AS $$

BEGIN

  NEW.updated_at = NOW();

  RETURN NEW;

END;

$$ LANGUAGE plpgsql;

Authentication Logic

Supabase provides built-in authentication that we'll leverage for the meditation app.

Authentication Methods

The app will support the following auth methods:

  1. Email/Password - Primary method with email verification
  2. Google OAuth - Social login option
  3. Apple Sign-In - Required for iOS apps

Auth Implementation

1. User Signup Flow

  1. User submits email/password or signs in with social provider
  2. On successful auth, a record is created in the auth.users table (handled by Supabase)
  3. A trigger function creates a corresponding entry in the public users table
  4. If a referral code was provided, it's processed to associate the user with a referrer

-- Create auth trigger to add new users to our users table

CREATE OR REPLACE FUNCTION public.handle_new_user()

RETURNS TRIGGER AS $$

BEGIN

  INSERT INTO public.users (id, email)

  VALUES (NEW.id, NEW.email);

  RETURN NEW;

END;

$$ LANGUAGE plpgsql SECURITY DEFINER;

 

-- Trigger the function every time a user is created

CREATE TRIGGER on_auth_user_created

  AFTER INSERT ON auth.users

  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

2. JWT-Based Session Management

Supabase handles JWT generation, validation, and refresh automatically. The app will:

  1. Store JWT tokens securely in the client using flutter_secure_storage
  2. Use Supabase client's built-in token refresh mechanism
  3. Leverage JWT claims for Row Level Security policies

3. Password Reset Flow

  1. User requests password reset from login screen
  2. Supabase sends a magic link to the user's email
  3. User clicks the link, which opens the app via deep link
  4. App validates the token and allows password reset

Row Level Security (RLS) Policies

RLS policies control data access at the row level, ensuring users can only access authorized data.

Enable RLS on All Tables

-- Enable RLS on all tables

ALTER TABLE users ENABLE ROW LEVEL SECURITY;

ALTER TABLE artists ENABLE ROW LEVEL SECURITY;

ALTER TABLE meditation_tracks ENABLE ROW LEVEL SECURITY;

ALTER TABLE meditation_sessions ENABLE ROW LEVEL SECURITY;

ALTER TABLE user_referrals ENABLE ROW LEVEL SECURITY;

ALTER TABLE artist_earnings ENABLE ROW LEVEL SECURITY;

User Table Policies

-- Users can only view and update their own profiles

CREATE POLICY "Users can view their own data" ON users

  FOR SELECT USING (auth.uid() = id);

 

CREATE POLICY "Users can update their own data" ON users

  FOR UPDATE USING (auth.uid() = id);

Artists Table Policies

-- Anyone can view artists

CREATE POLICY "Anyone can view artists" ON artists

  FOR SELECT USING (true);

 

-- Only admin users can modify artists (in a real app, implement admin roles)

CREATE POLICY "Only admins can modify artists" ON artists

  FOR ALL USING (is_admin_check());

Meditation Tracks Policies

-- Anyone can view tracks

CREATE POLICY "Anyone can view tracks" ON meditation_tracks

  FOR SELECT USING (true);

Meditation Sessions Policies

-- Users can only view their own meditation sessions

CREATE POLICY "Users can view their own sessions" ON meditation_sessions

  FOR SELECT USING (auth.uid() = user_id);

 

-- Users can insert their own meditation sessions

CREATE POLICY "Users can insert their own sessions" ON meditation_sessions

  FOR INSERT WITH CHECK (auth.uid() = user_id);

 

-- Users can update their own meditation sessions

CREATE POLICY "Users can update their own sessions" ON meditation_sessions

  FOR UPDATE USING (auth.uid() = user_id);

User Referrals Policies

-- Users can view their own referrals (as referrer or referee)

CREATE POLICY "Users can view referrals they're part of" ON user_referrals

  FOR SELECT USING (

    auth.uid() = user_id OR

    auth.uid() = referrer_id

  );

 

-- Users can create referrals where they are the referrer

CREATE POLICY "Users can create referrals they initiate" ON user_referrals

  FOR INSERT WITH CHECK (auth.uid() = referrer_id);

Storage Rules

Supabase Storage is used for artist images and meditation audio tracks.

Storage Buckets

Set up the following storage buckets:

  1. artist_images - Public bucket for artist profile photos
  2. audio_tracks - Private bucket for meditation audio content
  3. user_avatars - Private bucket for user profile pictures

Storage RLS Policies

1. Artist Images Bucket (Public)

-- Anyone can view artist images

CREATE POLICY "Anyone can view artist images"

ON storage.objects FOR SELECT

USING (bucket_id = 'artist_images');

 

-- Only admins can modify artist images

CREATE POLICY "Only admins can modify artist images"

ON storage.objects FOR INSERT

WITH CHECK (

  bucket_id = 'artist_images' AND

  is_admin_check()

);

2. Audio Tracks Bucket (Protected)

-- Anyone can access audio tracks (but they need to know the path)

CREATE POLICY "Anyone can access audio tracks"

ON storage.objects FOR SELECT

USING (bucket_id = 'audio_tracks');

 

-- Only admins can upload audio tracks

CREATE POLICY "Only admins can upload audio tracks"

ON storage.objects FOR INSERT

WITH CHECK (

  bucket_id = 'audio_tracks' AND

  is_admin_check()

);

3. User Avatars Bucket (Private)

-- Users can only access their own avatars

CREATE POLICY "Users can access their own avatars"

ON storage.objects FOR SELECT

USING (

  bucket_id = 'user_avatars' AND

  (storage.foldername(name))[1] = auth.uid()::text

);

 

-- Users can only upload to their own avatar folder

CREATE POLICY "Users can upload their own avatars"

ON storage.objects FOR INSERT

WITH CHECK (

  bucket_id = 'user_avatars' AND

  (storage.foldername(name))[1] = auth.uid()::text

);

Storage Path Structure

Organize files with the following path structure:

  1. Artist Images: /artist_images/{artist_id}/{filename}
  2. Audio Tracks: /audio_tracks/{artist_id}/{track_id}/{filename}
  3. User Avatars: /user_avatars/{user_id}/{filename}

API Access Patterns

The app interacts with Supabase using the following patterns:

Authentication API

// Sign up a new user

Future<UserModel?> signUp({

  required String email,

  required String password,

  String? referralCode,

}) async {

  final response = await supabaseClient.auth.signUp(

    email: email,

    password: password,

    data: referralCode != null ? {'referral_code': referralCode} : null,

  );

 

  // Process user record

  if (response.user != null) {

    // Process referral if provided

    if (referralCode != null) {

      await _processReferral(response.user!.id, referralCode);

    }

    return await _fetchUserProfile(response.user!.id);

  }

  return null;

}

 

// Sign in existing user

Future<UserModel?> signIn({

  required String email,

  required String password,

}) async {

  final response = await supabaseClient.auth.signInWithPassword(

    email: email,

    password: password,

  );

 

  if (response.user != null) {

    return await _fetchUserProfile(response.user!.id);

  }

  return null;

}

Data Access API

// Get all available artists

Future<List<ArtistModel>> getArtists() async {

  final response = await supabaseClient

      .from('artists')

      .select()

      .order('name');

 

  return (response as List)

      .map((json) => ArtistModel.fromJson(json))

      .toList();

}

 

// Get tracks for a specific artist

Future<List<TrackModel>> getTracksByArtist(String artistId) async {

  final response = await supabaseClient

      .from('meditation_tracks')

      .select()

      .eq('artist_id', artistId)

      .order('title');

 

  return (response as List)

      .map((json) => TrackModel.fromJson(json))

      .toList();

}

 

// Save a completed meditation session

Future<SessionModel?> saveSession({

  required String userId,

  required String trackId,

  required int duration,

  required bool completed,

}) async {

  final now = DateTime.now();

  final date = DateTime(now.year, now.month, now.day);

 

  final response = await supabaseClient

      .from('meditation_sessions')

      .insert({

        'user_id': userId,

        'track_id': trackId,

        'duration': duration,

        'completed': completed,

        'completed_at': now.toIso8601String(),

        'date': date.toIso8601String(),

      })

      .select()

      .single();

 

  if (response != null) {

    return SessionModel.fromJson(response);

  }

  return null;

}

Storage API

// Upload user avatar

Future<String?> uploadUserAvatar(String userId, Uint8List fileBytes, String fileName) async {

  final fileExt = fileName.split('.').last;

  final filePath = '$userId/avatar.$fileExt';

 

  final response = await supabaseClient

      .storage

      .from('user_avatars')

      .uploadBinary(

        filePath,

        fileBytes,

        fileOptions: FileOptions(

          contentType: 'image/$fileExt',

          upsert: true,

        ),

      );

 

  if (response.contains('error')) {

    return null;

  }

 

  // Get public URL

  final imageUrl = supabaseClient

      .storage

      .from('user_avatars')

      .getPublicUrl(filePath);

 

  // Update user profile with avatar URL

  await supabaseClient

      .from('users')

      .update({'avatar_url': imageUrl})

      .eq('id', userId);

 

  return imageUrl;

}

 

// Get meditation audio URL

Future<String?> getMeditationAudioUrl(String trackPath) async {

  try {

    return supabaseClient

        .storage

        .from('audio_tracks')

        .getPublicUrl(trackPath);

  } catch (e) {

    print('Error getting audio URL: $e');

    return null;

  }

}

Realtime Subscriptions

For features that benefit from realtime updates:

// Subscribe to user streak updates

void subscribeToUserStreakUpdates(String userId, Function(Map<String, dynamic>) callback) {

  final subscription = supabaseClient

      .from('users')

      .on(SupabaseEventTypes.update, callback)

      .eq('id', userId)

      .subscribe();

     

  // Store subscription reference for later cleanup

  _activeSubscriptions.add(subscription);

}

 

// Clean up subscriptions when no longer needed

void disposeSubscriptions() {

  for (final subscription in _activeSubscriptions) {

    subscription.unsubscribe();

  }

  _activeSubscriptions.clear();

}

Security Best Practices

  1. Never Disable RLS - Always use RLS policies to control data access
  2. Use WITH CHECK Clauses - For INSERT and UPDATE operations to validate data
  3. Avoid Direct ID Exposure - Use UUIDs instead of sequential IDs
  4. Implement JWT Validation - Validate claims for critical operations
  5. Encrypt Sensitive Data - Hash passwords and encrypt personal information
  6. Implement Rate Limiting - Protect against brute force attacks
  7. Regular Auditing - Review access logs and policy effectiveness

Deployment Considerations

  1. Environment Configuration
    • Store Supabase URL and API keys in a secure .env file
    • Use different Supabase projects for development and production
  2. Database Migrations
    • Use supabase migration commands to manage schema changes
    • Test migrations in a staging environment before production
  3. Backup Strategy
    • Configure automatic backups for the Supabase database
    • Test restore procedures periodically
  4. Monitoring
    • Set up alerts for unusual database activity
    • Monitor API usage and storage consumption

Troubleshooting and Common Issues

  1. RLS Policy Debugging
    • Use auth.uid() function carefully in policies
    • Test policies with different user contexts
  2. Storage Access Issues
    • Verify bucket permissions and folder structures
    • Check file paths match RLS policy patterns
  3. Authentication Errors
    • Confirm email verification settings
    • Verify JWT token expiration and refresh
  4. Performance Issues
    • Add appropriate indexes for frequent queries
    • Use count queries for large datasets to avoid loading full result sets
